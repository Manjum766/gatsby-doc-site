Release Notes Generator

Automated sprint release notes — pulls every Jira ticket from your active sprint, correlates merged PRs, and produces structured Markdown via GPT-4o.

Jira Project: SCRUM
Jira Board: manjum005.atlassian.net/jira/software/projects/SCRUM/boards/1
Workflow file: .github/workflows/release-notes.yml
Trigger: PR merged to master / manual dispatch

Overview

Every time a PR is merged to master — or on demand via the GitHub Actions UI — the Release Notes Generator workflow automatically:

  • Authenticates with your Jira instance and fetches all tickets for the active (or named) sprint on the SCRUM board
  • Collects all GitHub PRs merged since the last published release tag
  • Sends both datasets to GPT-4o which groups, summarises, and formats them into professional release notes
  • Creates a GitHub Release with the generated Markdown as its body
  • Opens a PR to update CHANGELOG.md — no manual editing needed
No OpenAI key? The workflow degrades gracefully — it produces a plain Markdown list of Jira tickets and PRs without LLM summarisation. Everything else still works.

How It Works — Step by Step

1
Trigger
PR merged into master, or manual dispatch from the GitHub Actions UI with an optional sprint name and release tag.
2
Resolve metadata
Computes the release tag (supplied or auto-generated as vYYYY.MM.DD-<sha>) and sprint identifier (active or named).
3
Collect merged PRs
Uses the GitHub CLI to list all PRs merged into master since the last published GitHub Release tag. Each PR title, author, labels, and first 500 chars of body are captured.
4
Fetch Jira sprint tickets
Calls the Jira Agile REST API to locate board SCRUM-1 on manjum005.atlassian.net, then fetches every issue in the target sprint — including summary, type, status, priority, assignee, and labels.
5
GPT-4o release notes
Both datasets are sent to GPT-4o with a structured prompt. The LLM groups issues by type (Story / Bug / Task), maps PRs to Jira tickets where keys match, writes a 2–3 sentence executive Highlights summary, and outputs clean Markdown.
6
Create GitHub Release
The generated Markdown is used as the GitHub Release body. If the tag already exists the release body is updated in-place; otherwise a new release is created pointing at master.
7
CHANGELOG PR
The new release notes are prepended to CHANGELOG.md on a new branch chore/release-notes-<tag> and a ready-to-merge PR is opened automatically.

Required Secrets Setup

Add these in GitHub → Settings → Secrets and variables → Actions → New repository secret:

JIRA_BASE_URLFull Atlassian URL — e.g. https://manjum005.atlassian.netRequired
JIRA_USER_EMAILYour Atlassian account email addressRequired
JIRA_API_TOKENAtlassian API token (Account Settings → Security → API tokens)Required
JIRA_PROJECT_KEYJira project key — SCRUM for this boardRequired
OPENAI_API_KEYOpenAI API key for GPT-4o. When absent, raw Markdown is generated insteadOptional

Getting a Jira API Token

  1. Go to id.atlassian.com → Security → API tokens
  2. Click Create API token, name it GitHub-Release-Notes
  3. Copy the token immediately — it won't be shown again
  4. Set JIRA_USER_EMAIL to the email of the account that created the token

Trigger Modes

Automatic — on PR merge

The workflow triggers on every PR that is merged to master. It uses the active sprint and auto-generates a date-based tag (vYYYY.MM.DD-<sha>).

Manual — workflow dispatch

Navigate to GitHub → Actions → Release Notes Generator → Run workflowand fill in the two optional inputs:

InputExampleDefault
sprint_nameSprint 1Active sprint on the board
release_tagv1.2.0Auto-generated: vYYYY.MM.DD-<sha>
Sprint name must be exact. The workflow matches the sprint name character-for-character against the Jira API. Check the exact name at manjum005.atlassian.net/jira/software/projects/SCRUM/boards/1 (Backlog view shows sprint names).

Jira Integration Details

API endpoints used

PurposeEndpoint
Find board ID for SCRUM project/rest/agile/1.0/board?projectKeyOrId=SCRUM

Get named sprint/rest/agile/1.0/board/{id}/sprint?maxResults=50

Fields captured per Jira ticket

  • key (e.g. SCRUM-12)
  • summary
  • issue type (Story / Bug / Task / Sub-task)
  • status (To Do / In Progress / Done)
  • priority
  • assignee display name
  • labels
  • direct URL to ticket
Sprint not found? If the sprint name doesn't match exactly, the workflow falls back to the 50 most recently updated issues in the project — so you always get something useful rather than an empty release.

One-Time Repository Setup

  1. Workflow permissions: Go to Settings → Actions → General → Workflow permissions and select Read and write. This allows the workflow to create releases and open PRs.
  2. Add all 5 secrets listed in the secrets table above.
  3. First run: Trigger manually via Actions → Release Notes Generator → Run workflow to validate the Jira connection before relying on the automatic merge trigger.
Verify Jira connectivity manually:
curl -u YOUR_EMAIL:YOUR_API_TOKEN \ "https://manjum005.atlassian.net/rest/agile/1.0/board?projectKeyOrId=SCRUM" \ | jq '.values[].name'
You should see the board name printed. If you get a 401, the email or token is wrong.

Troubleshooting

401 Unauthorized from Jira API

  • JIRA_USER_EMAIL must be the exact email of the account that owns the API token
  • API tokens expire — regenerate at id.atlassian.com → Security → API tokens
  • The Atlassian free plan requires the user to be a project member

Sprint not found / empty ticket list

  • Check the exact sprint name at SCRUM board → Backlog view
  • Sprint names are case-sensitive (e.g. Sprint 1 not sprint 1)
  • If no sprint is active and input is blank, the fallback lists recent project issues

GitHub Release already exists

The workflow calls gh release edit if the tag already exists, updating the body in-place. You can safely re-run the workflow after fixing secrets.

CHANGELOG PR conflicts

If a chore/release-notes-* branch already exists from a previous run, delete it from GitHub and re-run. Each run creates a fresh branch with the tag in the name.

LLM produces empty or truncated output

  • Verify OPENAI_API_KEY is valid and the account has GPT-4o access
  • Very large sprints (100+ tickets) may approach token limits — the workflow caps at 2000 output tokens
  • The Python fallback always produces output regardless of LLM availability
